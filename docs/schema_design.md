# 剧本 YAML Schema 设计文档

## 1. 设计目标

本 Schema 旨在为"AI 小说转剧本工具"提供一套**通用、可扩展、机器可读且人工可编辑**的剧本数据结构。设计时遵循以下核心原则：

| 原则 | 说明 |
|------|------|
| **语义完整** | 覆盖小说的主要叙事元素：角色、对话、动作、环境、心理 |
| **分层清晰** | 按"幕→场景→事件"三层结构组织，适配不同粒度需求 |
| **人工友好** | YAML 格式 + 英文键名 + 注释说明，降低学习成本 |
| **机器友好** | 结构化数据可被程序直接校验、渲染、转换 |
| **可扩展** | 预留灯光、音效、镜头等字段，不破坏现有结构 |

---

## 2. 为什么选择 YAML？

### 2.1 对比备选方案

| 格式 | 优势 | 劣势 |
|------|------|------|
| **YAML** | 可读性极强、支持注释、缩进结构天然适配剧本层级 | 缩进敏感、不适用于超大数据 |
| JSON | 解析速度快、工具生态成熟 | 无注释、大括号密集、人类编辑体验差 |
| XML | 强类型、Schema 验证 | 冗余标签多、可读性差 |
| TOML | 简洁、适合配置 | 不支持复杂嵌套、不适合长文本 |
| 纯文本 | 最自由 | 无结构化，无法程序化处理 |

**结论**：YAML 在"人机两读"上取得了最佳平衡——AI 可以稳定生成，人类编剧可以直接编辑修改。

### 2.2 剧本行业惯例

专业剧本格式（如 Fountain、Final Draft）采用纯文本 + 标签标记。本 Schema 是**结构化中间格式**，定位在 AI 输出与最终剧本格式之间，优势在于：

- 保留原始叙事元素的分类信息（对话 vs 动作 vs 旁白）
- 可程序化转换为 Fountain、HTML、PDF 等最终格式
- 支持增量修改和版本对比

## 3. 结构设计详解

### 3.1 顶层字段

```yaml
title       # 剧本标题 [必填]
characters  # 角色列表 [必填]
acts        # 幕的列表 [必填]
```

**设计原因**：剧本的最小"三要素"——**谁**（角色）、**什么故事**（幕）、**叫什么**（标题）。三者缺一不可。

### 3.2 角色表设计

```yaml
characters:
  - name: "林晓舟"
    description: "拾光咖啡馆店主，沉稳内敛"
```

**设计原因**：
- 角色信息集中在顶层，便于全局管理和预览
- 场景中通过 `characters_present` 仅引用角色名，避免数据冗余
- 预留 `age`、`personality`、`actor` 等字段的扩展空间

### 3.3 三层事件结构：Act → Scene → Event

这是本 Schema 的核心架构。

#### 为什么是三层？

| 层级 | 对应小说概念 | 功能 | 举例 |
|------|-------------|------|------|
| **Act**（幕） | 章 / 部分 | 最大叙事段落，通常对应情节的"起承转合" | 第一幕：危机降临 |
| **Scene**（场景） | 自然段/场景 | 时间+地点一致的连续事件 | 咖啡馆内，傍晚 |
| **Event**（事件） | 句/对话 | 最小叙事单元，不可再分 | 一句台词、一个动作 |

**设计原因**：
- **三层是人类叙事的自然粒度**：观众感知"幕"，导演规划"场景"，演员执行"事件"
- **适配 AI 分析能力**：当前 LLM 在处理"事件"级别（单句粒度）时最稳定
- **便于后期精编**：编剧可以直接在"事件"级别微调，而不需要重写整个场景

### 3.4 四种事件类型

| 类型 | 用途 | 示例 | 对应小说中的 |
|------|------|------|-------------|
| `dialogue` | 角色对话 | `"别擦了，杯子都要被你擦薄了。"` | 引号内对话 |
| `action` | 角色动作 | `林晓舟走到窗边` | 叙述性行为 |
| `stage_direction` | 环境描写 | `窗外的路灯忽然亮了` | 环境描绘 |
| `voiceover` | 内心独白/旁白 | `我想起两年前……` | 心理描写/回忆 |

#### 设计原因

**为什么不合并为一种通用事件类型？**

1. **区分度就是价值**：
   - 对话需要演员念出来 → 需要标注 `character` + `emotion`
   - 动作需要演员做出来 → 需要标注 `character` + `description`
   - 舞台指示是导演/舞美的工作 → 不需要 `character`
   - 旁白是画外音 → 特殊处理

2. **下游需求不同**：
   - 导出 Fountain 格式时，对话和动作有不同的前缀符号
   - 生成分镜脚本时，只关注 `action` 和 `stage_direction`
   - 统计台词量时，只需要 `dialogue` 类型

3. **让 AI 的任务更明确**：
   - 明确的类型标签减少了 LLM 的自由发挥空间
   - 每种类型的必填字段不同，降低了输出格式错误率

### 3.5 可选字段设计

```yaml
dialogue.emotion      # [可选] 情感标记
voiceover.character   # [可选] 默认为"旁白"
```

**设计原因**：
- **不强制填写**：不是每句台词都有明显的情感色彩，强制要求会导致"假情感"
- **保留扩展口**：编剧可以在精编阶段自行补充
- **AI 弹性**：LLM 在有明显情感时标记，无需猜测

## 4. 字段命名规范

| 规范 | 约定 | 原因 |
|------|------|------|
| 大小写 | 全小写 | Python/YAML 惯例，避免大小写不一致导致的 bug |
| 分隔符 | 下划线 `_` | YAML 键名中连字符可能有歧义 |
| 类型标记 | 无类型后缀 | 不做 `name_str`、`count_int` 式命名 |
| 复数 | 列表用复数 | `characters`、`events` 而非 `character_list` |

## 5. 未来扩展路径

### 5.1 已在注释中预留的字段

```
lighting       → 场景灯光设计
sound          → 环境音效/背景音乐
camera         → 镜头建议
transition     → 转场效果
duration       → 建议时长
```

### 5.2 预留的事件类型

```yaml
# FUTURE: type: "lighting_change"    — 灯光变化
# FUTURE: type: "sound_effect"       — 音效
# FUTURE: type: "music"              — 背景音乐
# FUTURE: type: "camera_move"        — 镜头运动
# FUTURE: type: "transition"         — 转场
```

### 5.3 扩展原则

1. **向后兼容**：新增字段不改变现有字段含义
2. **可选优先**：新字段以 optional 形式加入
3. **不破不立**：现有四种事件类型被证明足够前，不加新类型

## 6. 与竞品方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **本 Schema** | 专为 AI 转换设计、人机友好、分层清晰 | 需额外步骤转为最终剧本格式 |
| Fountain 格式 | 行业标准、纯文本 | AI 输出不稳定、缺少结构化角色信息 |
| 直接 LLM 输出纯文本 | 最简单 | 无结构、无法程序化处理 |
| JSON Schema | 强校验 | 人工编辑体验差 |

---

## 7. 使用流程

```
小说文本 → AI 分析 → YAML 剧本（本 Schema） → 人工精编 → 导出目标格式
                         ↑                          ↑
                   自动生成，初稿                  编剧在 YAML 中修改
```

**核心设计思想**：AI 做"粗活"（结构化初稿），人类做"细活"（打磨润色）。Schema 的目标是让 AI 的输出一次通过率尽可能高，减少人类返工成本。
